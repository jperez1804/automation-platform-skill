## Hostinger MCP gotcha — IMPORTANT

The Hostinger MCP cannot do container ops on this VM:

- `getProjectListV1` and `getProjectContainersV1` return:
  ```
  [VPS:2044] Currently installed operating system does not support Docker Manager
  ```
- The MCP **can** do: VM lifecycle (start/stop/recreate), firewall management, snapshots, DNS.
- The MCP **cannot** do: docker exec, docker logs, container listing, container restart, anything that needs Docker Manager.

**Practical implication:** for anything inside containers (running SQL queries against `n8n-client1-postgres`, restarting a tenant container, tailing logs, exec'ing a shell), **SSH is mandatory** — use `ssh vps` and run docker commands there. The MCP is useful only for VM-level ops, DNS, and firewall.

## Three deploy gotchas (check these first when Deploy misbehaves)

Each masquerades as "the workflow is broken." Going through them in order saves time. (Source: `feedback_vps_ops_gotchas.md`, surfaced during the 2026-05-02 theming rollout.)

### 1. `/opt/scripts/update-dashboards.sh` drift

The VPS copy is installed by `curl` from `main` and never auto-syncs. If the repo script changed (e.g. a compose-flag refactor) but nobody refreshed the VPS copy, the workflow runs the stale version.

**Diagnose:**
```bash
ssh vps 'sha256sum /opt/scripts/update-dashboards.sh'
# compare against:
sha256sum scripts/update-dashboards.sh   # in the dashboard repo on main
```

**Fix:** `deploy` can't create new files in `/opt/scripts/`, so refresh by overwriting in place:
```bash
ssh vps 'curl -fsSL https://raw.githubusercontent.com/jperez1804/botargento-dashboard/main/scripts/update-dashboards.sh -o /tmp/u.sh && cat /tmp/u.sh > /opt/scripts/update-dashboards.sh'
```

### 2. `VPS_SSH_KEY` secret malformed

Symptom: workflow fails in <5s with `ssh.ParsePrivateKey` or "no key found".

Cause: the secret paste lost newlines, or the public half was pasted instead of the private half.

**Fix:** rotate via a shell that preserves byte content:
```bash
gh secret set VPS_SSH_KEY < ~/.ssh/botargento_deploy
```
Then ensure the matching public key is in `/home/deploy/.ssh/authorized_keys` on the VPS.

### 3. fail2ban bans the GitHub runner IP

Symptom: `dial tcp …:22: i/o timeout` — TCP-level, not auth.

Cause: after back-to-back failed-auth retries, sshd's fail2ban jail can ban the runner IP.

**Fix (immediate):** retry the workflow once — runner IPs rotate.
**Fix (manual):** Jonatan SSHes in and `sudo fail2ban-client unban <ip>`.
**Long-term:** allowlist GitHub Actions IP ranges in fail2ban's `ignoreip`.

## Per-tenant n8n bootstrap gotchas

These surfaced the first time a brand-new tenant n8n was deployed alongside `client1` (Plec, 2026-05-08). The dashboard provisioning script (next section) handles its own gotchas, but a fresh **n8n + Postgres** stack on the VPS hits these three:

### 1. `/opt/n8n/` is `root:root`, not `deploy`

The parent directory is `drwxr-xr-x root:root`. `deploy` (the SSH user) cannot `mkdir /opt/n8n/<newtenant>/` — only the existing tenant subdirectories are deploy-owned, not the parent.

**Bootstrap step (manual, one-time per new tenant):** from a privileged session on the VPS (root SSH, or someone with sudo), create the directory and chown to deploy:

```bash
sudo mkdir -p /opt/n8n/<newtenant>
sudo chown deploy:deploy /opt/n8n/<newtenant>
```

After this, `deploy` (and therefore `scp` from the laptop's `ssh vps` alias) can write into it.

### 2. Use `docker-compose` (with hyphen), not `docker compose`

This VPS has **Docker 29.x** but the Compose plugin **is not installed** as a `docker` subcommand. Compose is available only as the standalone binary `/usr/local/bin/docker-compose` (v2.x, current).

```bash
# ✓ works
ssh vps 'cd /opt/n8n/<tenant> && docker-compose up -d'

# ✗ fails with: unknown shorthand flag: 'd' in -d
ssh vps 'cd /opt/n8n/<tenant> && docker compose up -d'
```

Same flag set as the plugin (it's just the binary form of v2). Affects `up`, `ps`, `logs`, `down`, etc. — every compose call.

### 3. n8n 2.x ignores `N8N_BASIC_AUTH_*` env vars

Legacy n8n 0.x/1.x used `N8N_BASIC_AUTH_ACTIVE`/`_USER`/`_PASSWORD` to gate the editor. **n8n 2.x replaced this with user management** — those env vars are silently ignored.

**Practical effect:** when you first hit `https://<tenant>.botargento.com.ar/` after `docker-compose up -d`, n8n shows the **owner-creation wizard**, not a basic-auth dialog. Create the first owner email + password there.

Leaving the legacy env vars in `.env` is harmless (they're parsed and discarded). If you want a clean .env, drop them — but new tenants tend to copy from `client1`, so they'll keep showing up.

## Tenant provisioning workflow

`scripts/provision-tenant.sh <clientN>` from the dashboard repo. Interactive bash runbook on Jonatan's laptop. Steps (target ≤15 min):

1. Verify n8n + Postgres exist for that tenant on the VPS
2. Generate strong DB password + `AUTH_SECRET`
3. Apply migrations to the tenant's Postgres
4. Collect: `CLIENT_NAME`, `CLIENT_PRIMARY_COLOR`, email auth credentials, allowlist of admin emails
5. Generate `dashboard.compose.yml` + `dashboard.env`, write them to `/opt/n8n/<clientN>/` (env at mode 0600)
6. Pull image from GHCR, start container
7. Append `<clientN>` to `/opt/scripts/tenants.txt` (so future `tenant=all` deploys include it)

This script also handles the first-time admin promotion (the SQL `UPDATE dashboard.allowed_emails SET role='admin' WHERE email=...`) so existing-tenant manual UPDATEs are only needed for **already-deployed** tenants.

## Operator one-liners (via `ssh vps`)

```bash
# List running containers
ssh vps 'docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"'

# Tail dashboard logs for a tenant
ssh vps 'docker logs -f --tail 200 client1-dashboard'

# Run psql against a tenant's Postgres
ssh vps 'docker exec -it n8n-client1-postgres psql -U dashboard_app -d client1'

# Inspect a container's env (sanity-check after a deploy)
ssh vps 'docker inspect client1-dashboard --format "{{range .Config.Env}}{{println .}}{{end}}"'

# Check VPS disk
ssh vps 'df -h /'
```
